# 設定項目メンテナンス方針

各 agent CLI の設定編集レジストリ（`SETTINGS_FIELDS`）と i18n を、各 CLI の公式仕様に追随して継続的に維持するための判定基準と手順を定める。設定項目の追加・廃止は各 CLI の更新に伴い随時発生するため、その都度この方針に従う。

対象レジストリの所在と型・保存の仕組みそのものは `システム仕様.md` の「設定編集レジストリ」を参照する。本書は「どのキーを登録／廃止し、どう型付けするか」の判定基準と作業手順を扱う。

## 1. 一次情報（公式ソース）

設定キーの存在・型・既定値・選択肢は、必ず以下の一次情報で確認する。記憶や二次情報に頼らない。

| agent | 参照先 |
| --- | --- |
| Claude Code | 設定リファレンス `https://code.claude.com/docs/en/settings.md`。マネージド専用は `https://code.claude.com/docs/en/permissions.md` の「Managed-only settings」表。 |
| Codex CLI | config リファレンス `https://learn.chatgpt.com/docs/config-file/config-reference`（`developers.openai.com/codex/config-reference` のリダイレクト先）。 |
| Grok（Grok Build） | 公式ドキュメント `https://docs.x.ai/build/settings/reference` と公式ソース `github.com/xai-org/grok-build`（`crates/codegen/xai-grok-shared/src/ui_config.rs` の `UiConfig` が `[ui]` の正典、`.../xai-grok-pager/src/settings/defs.rs`、`.../xai-grok-pager/docs/user-guide/05-configuration.md`）。 |
| Antigravity CLI（Agy） | `https://antigravity.google/docs/cli/settings`。設定ファイルは `~/.gemini/antigravity-cli/settings.json`。Gemini CLI と領域を共有するが、Antigravity 公式で裏付けの取れないキーは登録しない。 |
| OpenCode | 公式 JSON スキーマ `https://opencode.ai/config.json`（正典）と `https://opencode.ai/docs/config`。 |

## 2. 登録するキーの条件

次をすべて満たす「ユーザーが自分の設定ファイルに書けるキー」だけを登録する。

- 公式の設定リファレンス／スキーマ／ソースに存在する。
- マネージド専用でない（§3）。
- 非推奨（deprecated）・レガシー別名でない。正規キーのみを登録する。
- 対象 CLI の設定ファイル自体のキーである。別ファイル（キーバインド、`pager.toml`、`tui.json` 等）は対象外。キーバインド設定は本アプリの対応機能外とする。
- サーバーからフェッチされるリモート設定専用フィールドでない（例: Grok の `RemoteSettings`）。

## 3. マネージド専用の除外

組織管理者が配布する管理ポリシー設定でのみ有効なキーは、既存・新規を問わず登録しない。製品ごとの該当パターン:

- **Claude**: `permissions.md` の「Managed-only settings」表に載るキー（`minimumVersion`、`strictKnownMarketplaces`、`allowManaged*` 系、`sandbox.*.allowManaged*Only` など）。
- **OpenCode**: `enterprise`（central config）配下のキー。
- **Grok**: `RemoteSettings` 由来のリモート／管理フィールド（`managed_mcps.enabled` 等）、および `auth` / `grok_com_config.oidc` などの外部認証・enterprise 節。
- **Codex**: 現状該当なし（`config.toml` は個人設定）。

## 4. 型の割り当て

- 単一スカラー（boolean / string / number / 厳密 enum）→ 編集可能型（`boolean` / `string` / `number`。Claude の `env` 内フラグは `envFlag`）。
- 配列・マップ・動的テーブル・union・ハイブリッド（boolean | table）→ `directEdit`（一覧表示のみ。値欄に「ファイルを直接編集」を表示）。
- ハイブリッド（boolean | table）型は、サブスカラーを個別行に分解しない（親を `directEdit` にする）。分解すると boolean と table の書き込みが競合しうるため。例: Codex の `features.network_proxy` / `approval_policy` / `tools.web_search`。
- 親オブジェクトを `directEdit` で登録済みなら、その配下キーは登録済み扱いとし、個別追加しない。
- enum は、厳密な列挙値なら `choices` のセレクト。モデル ID・テーマ・プロバイダ ID など拡張余地のある値は `allowCustom: true` を付ける。
- 既定値は、公式で固定既定を確認できた場合のみ `defaultOn` / `defaultValue` を宣言する。環境依存・モデル依存・未記載の項目は宣言しない。

## 5. 廃止（削除）の条件

- 公式のリファレンス／スキーマ／ソースに存在しない（廃止済み・誤登録）。
- マネージド専用と判明した（§3）。

削除時は、空になったグループの i18n グループラベル（`SETTINGS_GROUP_ORDER` を持つ agent はその要素も）を併せて整理する。

## 6. 実装手順（追加・削除ともに）

1. 対象 agent の `shared/agents/<agent>/constants.ts` の `SETTINGS_FIELDS` を編集する。
2. i18n を日本語・英語の両方で編集する（`renderer/i18n/locales/<lang>/<agent>.ts` の `field.<key>` に、項目固有の短い label と desc）。`directEdit` 項目も i18n を必須とし、キーだけのタイトルや汎用説明へのフォールバックは使わない。i18n を欠くと UI にキー文字列がそのまま表示される。
3. `yarn lint`（`tsc --noEmit`）を通す。i18n の欠落は型エラーにならないため、追加した各キーが `constants.ts`・`ja`・`en` の 3 箇所すべてに揃っているかを grep 等で確認する。
4. `CHANGELOG.md` にエージェント別の増減件数で記載する（個々の項目名は不要）。例: `Added — Claude +5 / Codex +10 …`、`Removed — Claude -1 / …`。

## 7. 検証上の注意

- 巨大な公式ページ（特に Claude の `settings.md`）は、取得ツールのサマライザがページ後半を取得できず、存在しないキーを捏造して返すことがある。逐語取得・アルファベット範囲の分割取得・キーの個別存在確認で裏取りし、確証が取れないキーは追加しない。存在しないキーの誤登録は、削除漏れよりも有害である。
- 判断がつかないキーは追加せずに保留し、根拠を添えて記録する。
