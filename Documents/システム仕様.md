# Agent CLI DevKit システム仕様

## 1. 概要

Agent CLI DevKit は、AI エージェント CLI（Claude Code / Codex CLI / Grok CLI など）の開発環境を管理する Electron 製デスクトップアプリケーションである。
Claude Developer Tool と Codex Developer Tool の全機能を統合し、さらに Grok CLI（Grok Build）対応を追加したもので、UI・内部構造ともに「agent（Claude / Codex / Grok / 今後追加される CLI）」を単位としてグループ化されている。

- プロジェクト名: `agent_cli_devkit`
- 設定ファイルディレクトリ: `~/.agent_cli_devkit`（`src/shared/constants.ts` の `APP_DIR_NAME`。現状アプリ自体は永続設定を持たないため予約）
- 実行ファイル名: `agent_cli_devkit`
- 配布ファイル名: `agent-cli-devkit-<version>-<os>-<arch>`
- renderer へ公開するグローバル API: `window.agentCliDevkit`

## 2. アーキテクチャ

### 2.1 設計原則

1. **agent 別ファイル分離** — Claude 系と Codex 系の実装が同一ファイルに混在しないよう、層ごとに `claude/` / `codex/` ディレクトリへ分割する。両者を同時に参照してよいのは「登録ポイント」（registry・合成ファイル）のみ。
2. **共通化** — agent に依存しない処理（WSL 検出、ホームディレクトリのファイル操作、git 実行、サイズ計測、frontmatter/YAML/TOML 編集）は共通モジュールに置き、各 agent から利用する。
3. **拡張容易性** — 新しい agent CLI は「agent 別ディレクトリを追加し、各層の登録ポイントに 1 行追加する」だけで組み込める（§6 参照）。

### 2.3 プロセス構成と IPC

- メインプロセス起動時に `registerIpcHandlers()`（`src/main/ipc/index.ts`）が `WslDetector` と `AgentCliRunner` を 1 つずつ生成し、各 agent のマネージャに注入して IPC ハンドラを登録する。
- preload は `sandbox: false`（`contextIsolation: true` は維持）で動作し、agent 別ブリッジファイル（`preload/agents/*.ts`）を require 合成して `window.agentCliDevkit` を公開する。sandbox を無効化しているのは preload を agent 別ファイルに分割するためで、shared の定数・型を preload から直接 import できる利点もある。
- IPC チャンネルは agent プレフィックスで名前空間化されている:
  - Claude: `claude-desktop:*` / `claude-code:*` / `claude-cleanup:*` / `claude-asset:*` / `claude-settings:*` / `claude-plugin:*`
  - Codex: `codex-mcp:*` / `codex-asset:*` / `codex-cleanup:*` / `codex-config:*` / `codex-plugin:*`
  - Grok: `grok-mcp:*` / `grok-asset:*` / `grok-cleanup:*` / `grok-config:*` / `grok-plugin:*`
  - 共通: `window:*` / `system:*` / `updater:*` / `main:console`

### 2.4 renderer API（window.agentCliDevkit）

```text
window.agentCliDevkit
├── system   { getTheme, getLocale, getVersion, openExternal }
├── window   { minimize, maximize, close, isMaximized }
├── updater  { getState, check, download, quitAndInstall, onStateChanged }
├── claude   { desktop, code, cleanup, asset, settings, plugin }   # ClaudeIpcApi
├── codex    { mcp, asset, cleanup, config, plugin }               # CodexIpcApi
└── grok     { mcp, asset, cleanup, config, plugin }               # GrokIpcApi
```

### 3.4 共通機能

- **ダッシュボード** — agent ごとにグループ見出し（Claude / Codex / Grok）を表示し、配下に全機能のカードを並べる。カードはアイコンを左・テキストを右に置く横並びのコンパクトレイアウト。`renderer/agents/index.ts` の registry から自動生成。
- **タイトルバー** — ダッシュボードアイコンに続き、agent グループの代表ボタン（Claude / Codex / Grok）のみを表示するアコーディオン方式。代表ボタンをクリックするとそのグループの機能アイコンが展開され、別グループを展開すると先に開いていたグループは自動で閉じる（同じボタンの再クリックで閉じる）。画面遷移時は遷移先の機能が属するグループを自動展開する。バーガーメニューでは agent 名の小見出し付きで全機能を一覧。テーマ切替（ライト/ダーク）・言語切替（日本語/英語）・終了。
- **自動アップデート** — GitHub Releases を対象に electron-updater でチェック/ダウンロード/適用。
- **WSL 対応** — Windows では WSL distro を検出し、対象 CLI が導入された distro を各機能の管理対象として表示。ファイルアクセスは `\\wsl.localhost\` / `\\wsl$\` の UNC を優先し、不可なら wsl コマンド実行にフォールバック。
- **Agent・Skill の ZIP ダウンロード** — 既定ファイル名は `<agent>_<kind>[-<distro>].zip`（例: claude_skills.zip / grok_agents-Ubuntu-24.04.zip）。
- **Agent・Skill 一覧の参照列** — 各行に「ヘッダ」（frontmatter のみ）と「全体」（ファイル内容全体。agents=ファイル本体 / skills=SKILL.md。IPC `<agent>-asset:read-entry`）の 2 つの参照ボタンを 2 行で表示する。
- **設定の直接編集ダイアログ** — ダイアログを固定高（85vh）とし、スクロールはテキストエリア内の 1 本のみ（ネストスクロールなし）。
- **プラグイン管理（共通実装）** — 画面本体は agent 非依存の `renderer/components/plugins/PluginManagerView`（タブ: インストール済み / マーケットプレイス、プラグイン追加・マーケットプレイス追加ダイアログ、インストール前の信頼確認）で、各 agent の画面は API と文言キーを渡す薄いラッパー。機能差（個別リポジトリ直接インストールの可否、リモート marketplace.json URL の可否）は `shared/agents/<agent>/constants.ts` の `PLUGIN_CAPABILITIES` に宣言し、main 側が `PluginEnvReport.capabilities` として renderer へ渡す（renderer は OS 依存の constants を import しない）。変更操作は `AgentCliRunner` による agent CLI のヘッドレス実行（native は実行ファイル解決 + `.cmd`/`.bat` は `cmd.exe /d /s /c` 経由、WSL は `wsl.exe -d <distro> -- bash -lc`。引数は引用符・制御文字を拒否する検査つき）。一覧は CLI の `--json` 出力を共通型（`PluginEnvReport` / `PluginCatalogReport`）へ正規化する。インストールスコープは常に user。詳細は §3.5 参照。
- **i18n** — 日本語/英語。共通文言は `locales/<lang>/app.ts`、agent 固有文言は `locales/<lang>/<agent>.ts` に分離し、`claude.*` / `codex.*` の名前空間で参照する。

### 3.5 プラグイン管理の CLI 連携仕様

各 agent CLI のプラグイン仕様（2026-07-13 に実機で検証。Claude Code 2.1.207 / codex-cli 0.142-0.144 / grok 0.2.93）。

**実行するコマンド（マネージャごとの対応表）:**

| 操作 | Claude | Codex | Grok |
| --- | --- | --- | --- |
| インストール済み一覧 | `claude plugin list --json` | `codex plugin list --json` | `grok plugin list --json` |
| カタログ | `claude plugin list --available --json` | `codex plugin list --available --json` | `grok plugin list --available --json` |
| マーケットプレイス一覧 | `claude plugin marketplace list --json` | `codex plugin marketplace list --json` | `grok plugin marketplace list --json` |
| インストール | `claude plugin install <p>@<market>` | `codex plugin add <p>@<market> --json` | `grok plugin install <p>@<qualifier> --trust` |
| 個別リポジトリから | 非対応（2 段階方式） | 非対応（2 段階方式） | `grok plugin install <src> --trust`（@ref / #subdir 可） |
| アンインストール | `claude plugin uninstall <id>` | `codex plugin remove <id> --json` | `grok plugin uninstall <name> --confirm` |
| 有効/無効切り替え | `claude plugin enable\|disable <id>` | サブコマンド無し → config.toml の `[plugins."<id>"] enabled` を直接編集（対象行のみ） | `grok plugin enable\|disable <name>` |
| マーケットプレイス追加 | `claude plugin marketplace add <src>` | `codex plugin marketplace add <src> --json` | `grok plugin marketplace add <src>` |
| マーケットプレイス削除 | `claude plugin marketplace remove <name>` | `codex plugin marketplace remove <name>` | `grok plugin marketplace remove <name>` + フォールバック |

**ソース種別と互換性:**

- マーケットプレイスソース: Claude = owner/repo / Git URL（`#ref` 可）/ ローカルパス / リモート marketplace.json URL。Codex = ローカルパス / `owner/repo[@ref]` / HTTPS・SSH Git URL（URL 直接指定は不可）。Grok = Git URL / owner/repo / ローカルディレクトリ。
- **3 CLI とも Claude 形式のマニフェスト（`.claude-plugin/marketplace.json` + `plugin.json`）を読める**（実機検証済み）。このため「個別リポジトリからの追加」の 2 段階方式（マーケットプレイスとして追加 → 含まれるプラグインを選択）は同一リポジトリで 3 CLI 共通に運用できる。
- 組み込みマーケットプレイス（削除不可扱い）: Claude = `claude-plugins-official`、Codex = `openai-bundled` / `openai-primary-runtime` / `openai-curated`、Grok = `xAI Official`。定義は各 `constants.ts` の `*_BUILTIN_MARKETPLACES`。

**データ配置（表示は CLI 出力を正とし、以下は参考・フォールバック用）:**

- Claude: 実体 `~/.claude/plugins/cache/<market>/<plugin>/<version>/`、登録 `~/.claude/plugins/known_marketplaces.json` / `installed_plugins.json`、有効状態 `~/.claude/settings.json` の `enabledPlugins`（`{"<p>@<m>": bool}`）と `extraKnownMarketplaces`。
- Codex: `~/.codex/config.toml` の `[marketplaces.<name>]`（`source_type` / `source` / `last_updated`）と `[plugins."<p>@<m>"] enabled`。実体 `~/.codex/plugins/cache/`。`marketplace list` には config.toml に無い組み込みも出るため config 解析では不足する。
- Codex のカタログ補完: `plugin list --available --json` には description が含まれないため、各エントリの `source.path`（マーケットプレイススナップショット内のプラグイン実体）にある `.codex-plugin/plugin.json` から description / homepage を best-effort で補完する（native は直接読み、WSL は `\\wsl.localhost` / `\\wsl$` の UNC で読む。全エントリ並列で数百 ms 程度）。
- Grok: `~/.grok/config.toml` の `[[marketplace.sources]]`（ローカル= `path`、Git= `git` キー）と `[plugins] enabled = [...]`。実体 `~/.grok/installed-plugins/`。Claude の `known_marketplaces.json` を自動読込して一覧へ合成する（config.toml に無いソース = 他ツール由来として削除不可表示）。

**カタログのリンクと選択可否:**

- カタログ各エントリはリポジトリ URL（`homepage`）を持ち、追加ダイアログのプラグイン名直後の
  外部リンクアイコン（ラベルなし）のクリックで OS 既定ブラウザで開く
  （`system:open-external`。http/https のみ許可）。
  導出方法: Claude = カタログの `source`（github は repo、git-subdir は URL + `/tree/<ref>/<path>`）、
  Codex = plugin.json の `homepage` / `interface.websiteURL` / `repository`、
  Grok = プラグイン単位の URL が無いためマーケットプレイスの Git URL で代用。
- `PluginMarketplaceEntry.selectable` はプラグイン追加ダイアログでインストール元として
  選択可能かを示す。Grok では Claude 由来の外部マーケット（`origin='external'`）のうち
  公式 `claude-plugins-official`（`GROK_SELECTABLE_EXTERNAL_MARKETPLACES`）のみ true とし、
  ユーザーが Claude 用に追加した外部マーケットは Grok のインストール候補・カタログから除外する。

**Grok 固有の注意（0.2.93 実機で確認した不具合と対処）:**

1. マーケットプレイス指定子は表示名ではなく qualifier（ローカル= `local/<name>`、Git= `owner/repo`）。`GrokPluginManager.catalog()` が marketplace list の kind / URL から組み立てる。
2. `marketplace remove` はローカルソースを削除できない（exit 126）→ 失敗時に config.toml の該当 `[[marketplace.sources]]` ブロックを直接除去するフォールバックを実装。
3. `uninstall` 後に `[plugins] enabled` 配列へ名前が残ることがある → 成功後に該当エントリを掃除する。

**タイムアウト:** 一覧 60 秒 / 変更操作 180 秒（Claude 内部の git タイムアウト既定 120 秒
`CLAUDE_CODE_PLUGIN_GIT_TIMEOUT_MS` より長く設定）。private リポジトリの取得は各 CLI が参照する
`GITHUB_TOKEN` / `GITLAB_TOKEN` / `BITBUCKET_TOKEN` 等の環境変数に依存する。

## 4. 設定編集レジストリ

- Claude: `shared/agents/claude/constants.ts` の `SETTINGS_FIELDS`（`~/.claude/settings.json` の編集対象項目）。envFlag 型（env 内キーの ON/OFF）に対応。
- Codex: `shared/agents/codex/constants.ts` の `SETTINGS_FIELDS`（`~/.codex/config.toml` のドット区切りキー）。
- Grok: `shared/agents/grok/constants.ts` の `SETTINGS_FIELDS`（`~/.grok/config.toml` のドット区切りキー。公式 settings リファレンス準拠）。
- いずれも registry へ 1 項目追加すると読み取り・テーブル編集・保存・UI 表示まで反映される。関係のない既存項目には触れない差分マージ方式。

## 5. データモデル

本アプリはデータベースを使用しない。操作対象は各 CLI ツールの設定ファイル（JSON / TOML / YAML）とディレクトリのみで、独自の永続データは持たない。

## 6. agent 追加ガイド

新しい agent CLI（例: `gemini`）を追加する場合、以下の「agent 別ディレクトリの追加 + 登録ポイントへの 1 行追加」で完結する。

1. **shared** — `src/shared/agents/gemini/{constants,types,ipc}.ts` を作成。
   - 登録: `src/shared/ipc.ts` の `IpcApi` に `gemini: GeminiIpcApi;` を追加。
2. **main services** — `src/main/services/gemini/` にマネージャを実装。共通処理は `services/common/`（`WslDetector.getDistrosWithTest()` / `HomeFs`）と `main/utils/` を利用。
3. **main ipc** — `src/main/ipc/gemini/` にハンドラと `index.ts`（`registerGeminiIpcHandlers`）を作成。
   - 登録: `src/main/ipc/index.ts` に 1 行追加。
4. **preload** — `src/preload/agents/gemini.ts` にブリッジを作成。
   - 登録: `src/preload/index.ts` の api に `gemini: geminiApi,` を追加。
5. **renderer** — `src/renderer/agents/gemini/` に画面コンポーネントと `index.tsx`（`AgentModule` 定義: ルート/アイコン/文言キー）を作成。
   - 登録: `src/renderer/agents/index.ts` の `AGENT_MODULES` に 1 行追加。ダッシュボード・タイトルバー・ルートは自動反映される。
6. **i18n** — `src/renderer/i18n/locales/{ja,en}/gemini.ts` を作成し、各 `index.ts` に 1 行追加。文言キーは `gemini.*` 名前空間。

## 7. ビルド・配布

- 開発: `yarn dev`（tsc watch + vite + electron。renderer は BrowserRouter / <http://localhost:3001>）
- 検証: `yarn lint`（tsc --noEmit）/ `yarn eslint`
- 配布: `yarn dist:win|mac|linux`（renderer は HashRouter / `dist/renderer/index.html`）
- リリース: `yarn release:win|mac|linux`（GitHub ドラフトリリースへアップロード）

## 8. セキュリティ上の注意

- BrowserWindow は `contextIsolation: true` / `nodeIntegration: false` を維持しつつ、preload の agent 別ファイル分割のために `sandbox: false` としている。renderer に公開する API は `window.agentCliDevkit` 経由の明示的なブリッジのみ。
- クリーンアップ・削除操作はすべて既定 OFF のチェックボックス + 確認ダイアログを経る。使用中ファイルは best-effort でスキップし、結果として報告する。
